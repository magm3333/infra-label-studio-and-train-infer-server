ENDPOINTS PRINCIPALES

GET /status: estado global, GPU y configuracion default de entrenamiento.
GET /: UI web para jobs, lanzamiento de entrenamiento, estado, metricas, curvas, modelos entrenados.
POST /setup: Label Studio envia label_config; parsea nombres de labels y campos.
POST /predict: genera predicciones YOLO para tareas de Label Studio.
POST /train: lanza entrenamiento async para un proyecto Label Studio.
GET /api/jobs: lista jobs.
GET /api/jobs-fragment: fragmento HTML de cards de jobs para refresco en UI.
GET /api/jobs/<job_id>: detalle de un job.
DELETE /api/jobs/<job_id>: elimina job y artefactos asociados si no esta corriendo.
POST /api/jobs/<job_id>/cancel: cancela job corriendo reiniciando el backend con _os._exit(0).
GET /api/models: lista modelos entrenados y modelos disponibles.
POST /api/active-model: cambia el modelo activo de inferencia.
GET /download/models/<name>: descarga modelos .pt guardados.

ESTRUCTURA DE DATOS EN /app/data
Dentro del contenedor /app/data esta montado desde ./mydata/yolo-backend en el host.
Directorios/archivos usados:
/app/data/jobs.json: persistencia de jobs publicos.
/app/data/training/: datasets temporales convertidos a YOLO por entrenamiento.
/app/data/runs/: runs de Ultralytics con results.csv, pesos, graficos, etc.
/app/data/models/: modelos best.pt copiados al terminar entrenamiento.
Metadata de modelo: se guarda como archivo JSON junto al .pt, con path <modelo>.pt.json.

CONFIGURACION LABEL STUDIO ESPERADA
El proyecto Label Studio debe usar RectangleLabels sobre Image. Ejemplo minimo del README:
<View>
  <Image name="image" value="$image"/>
  <RectangleLabels name="label" toName="image">
    <Label value="precinto"/>
  </RectangleLabels>
</View>
Los nombres de labels deben coincidir con las clases del modelo YOLO, o con las clases definidas en el label_config parseado.

FLUJO DE INFERENCIA
Label Studio llama /setup y /predict.
parse_label_config extrae from_name, to_name, value y labels desde el XML de Label Studio.
prediction_for_task resuelve la imagen, ejecuta YOLO sobre ella y transforma cajas a formato RectangleLabels en porcentajes para Label Studio.
resolve_image_path puede resolver rutas locales bajo /label-studio/data, rutas relativas, o descargar temporalmente URLs si hace falta.

FLUJO DE ENTRENAMIENTO
POST /train recibe project/project_id/project_id_from_task o project desde task.
request_train_config arma la config efectiva desde payload, headers X-Train-* o defaults.
train_async corre en thread daemon.
Pasos de train_async:
1. Lee configuracion del proyecto desde Label Studio con /api/projects/<id>.
2. Parsea label_config.
3. Exporta tareas/anotaciones desde /api/projects/<id>/export?exportType=JSON.
4. Convierte anotaciones a dataset YOLO temporal con convert_to_yolo_dataset.
5. Lanza Ultralytics YOLO.train con data.yaml generado.
6. Copia weights/best.pt a /app/data/models/project-<project_id>-<jobid>-best.pt.
7. Carga el nuevo modelo como activo si termino bien.
8. Persiste metadata del modelo y jobs.json.

SPLIT TRAIN/VALID Y DATASET TEMPORAL
Cambio reciente importante: antes se usaba train=images/train y val=images/train, o sea validacion reutilizaba exactamente las mismas imagenes de entrenamiento.
Ahora existe un slider en la UI despues de Workers para elegir porcentaje de train.
Default: 70% train y 30% valid.
Backend: config key train_percent, default TRAIN_SPLIT_PERCENT=70. Se acepta por payload train_percent, alias train-percent o header X-Train-Train-Percent.
El porcentaje se clamp-ea entre 1 y 100.

Como se implementa sin modificar almacenamiento fisico de Label Studio:
YOLO/Ultralytics permite que train y val en data.yaml apunten a manifiestos .txt con rutas de imagenes.
YOLO NO acepta un manifiesto independiente de labels. Deriva las labels por convencion desde la ruta de imagen: images/... -> labels/... con mismo stem y extension .txt.
Por eso el backend crea un dataset temporal dentro de /app/data/training/project-<id>-<timestamp>/ con esta estructura:
images/all/: symlinks a imagenes originales si la imagen existe como archivo local estable. Si la imagen fue descargada temporalmente desde URL remota, se copia al dataset porque no hay ruta estable para symlink.
labels/all/: labels YOLO generadas desde las anotaciones RectangleLabels de Label Studio.
train.txt: manifiesto con rutas absolutas a imagenes/symlinks de train.
val.txt: manifiesto con rutas absolutas a imagenes/symlinks de validacion.
data.yaml: contiene path, train=<train.txt absoluto>, val=<val.txt absoluto>, names.

Esto NO modifica las imagenes originales de Label Studio. Solo crea artefactos temporales propios del entrenamiento en /app/data/training. Las labels YOLO son generadas porque Label Studio no las guarda nativamente en layout YOLO.

Split deterministico:
convert_to_yolo_dataset crea samples con hash sha256 de project_id, task id, image_value y nombre target. Ordena por hash y corta segun train_percent. Esto evita que el split cambie aleatoriamente entre ejecuciones con el mismo conjunto.

Conteos guardados:
dataset_info se guarda en el job y metadata del modelo con:
images: total de imagenes fisicas anotadas validas usadas.
train_percent.
val_percent.
train_images: cantidad fisica de imagenes de entrenamiento.
val_images: cantidad fisica de imagenes de validacion.
train_manifest.
val_manifest.
image_storage: "symlink_manifest".

Caso 100% train:
Si train_percent=100, val_images=0 y train_async pasa val=False a YOLO.train para evitar falla por manifiesto de validacion vacio.

UI WEB
La UI esta embebida como HTML/CSS/JS dentro de app.py, funcion index().
Secciones principales:
Jobs.
Lanzar entrenamiento.
Modelo activo para inferencia.
Estado.
Metricas YOLO del job.
Curvas de entrenamiento.
Best y paciencia.
Modelos entrenados.

Cambios recientes de UI:
Despues del input Workers se agrego slider Train split.
Muestra en vivo: X% train / Y% valid.
Al enviar el form convierte train_percent a Number y lo manda a /train.
El notice de envio incluye split X/Y.
Las cards de jobs muestran Train <n> / Valid <n> (<percent>%).
Las cards de jobs muestran Best epoca y, cuando existe, el valor mAP50-95 del Best.
La card Estado muestra Imagenes train, Imagenes valid y Split train del job seleccionado o modelo seleccionado.
El listado de Modelos entrenados muestra una columna mAP50-95 con el best_metric guardado en metadata, o n/d si no hay metrica.
El formulario Lanzar entrenamiento persiste sus valores en localStorage del browser con clave yolo11.trainForm y los restaura al recargar la pagina; si no hay valores guardados usa los defaults del HTML/backend.
En Metricas YOLO del job, la metrica mAP50-95 se renderiza con clase metric-highlight para que sobresalga visualmente.
El campo Proyecto Label Studio en Lanzar entrenamiento es un select alimentado desde /api/projects de Label Studio. Muestra id y titulo logico del proyecto. Si no se pueden cargar proyectos, muestra una opcion deshabilitada y un aviso; el entrenamiento requiere seleccionar un proyecto existente. La restauracion desde localStorage solo aplica si el proyecto guardado todavia existe entre las opciones.
El selector Modelo base incluye modelos externos y modelos entrenados. Los modelos externos se muestran siempre, arriba de la lista, y la fuente se etiqueta como externo. Los modelos entrenados se filtran por el Proyecto Label Studio seleccionado. Todas las opciones del selector muestran tamano en MB. Al lado del selector hay carga de archivo .pt para subir un modelo externo; se guarda en /app/data/external-models y no aparece en la tabla Modelos entrenados.
El formulario Lanzar entrenamiento incluye parametros avanzados recomendados: lr0, weight_decay y cos_lr. Se guardan en train_config, se persisten en localStorage con el resto del formulario y se pasan directo a YOLO.train como lr0, weight_decay y cos_lr.

METRICAS Y MODELOS
public_job agrega metrics si hay run_dir, leyendo results.csv con yolo_metrics.
training_artifact_summary calcula existencia/path de best.pt/last.pt, mejor mAP50-95, best_epoch, current_epoch y paciencia restante.
write_model_metadata guarda public_job(job) + saved_at junto al .pt.
read_model_metadata intenta leer <modelo>.json; si no existe, inferred_model_metadata intenta inferir desde nombre project-*-best.pt y run_dir.
model_option_label muestra nombre, fuente, fecha y mAP50-95 si hay metadata.
Los modelos entrenados se asocian a proyecto mediante metadata.project. Si un modelo entrenado antiguo no trae metadata.project, se infiere desde el nombre project-<id>-...; como fallback para modelos legacy se asume proyecto 19. Los modelos entrenados existentes al momento de este cambio ya tenian project: 19 en sus JSON.
Los modelos externos viven en /app/data/external-models, se listan solo como alternativas de modelo base/activo y no se incluyen en model_files(), por lo tanto no aparecen en la tabla de modelos entrenados.

CONSIDERACIONES Y RIESGOS
No usar comandos destructivos como git reset --hard o git checkout -- sin aprobacion explicita.
El worktree puede estar sucio; no revertir cambios ajenos.
La UI esta en un string HTML grande dentro de app.py; revisar cuidadosamente llaves dobles {{ }} por ser f-string de Python con JavaScript embebido.
Si se agregan variables al compose, recordar que app.py tiene default pero compose podria no exponerlas. Actualmente TRAIN_SPLIT_PERCENT tiene default en app.py y no fue agregado al docker-compose.yml.
El servicio usa image sin build en docker-compose.yml; docker compose up --build no reconstruye automaticamente esta imagen. Hay que ejecutar docker build manualmente si se quiere actualizar la imagen.
Como app.py esta montado por volumen, el contenedor ejecuta el app.py del host, no necesariamente el copiado en la imagen.
Los datasets temporales bajo /app/data/training se eliminan cuando se elimina un job mediante DELETE /api/jobs/<id>, porque delete_job_artifacts borra el directorio padre de dataset_yaml.
No hay tests automatizados dedicados para el split; la verificacion realizada fue compilacion Python y arranque Docker.
